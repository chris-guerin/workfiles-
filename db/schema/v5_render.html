v5 unifies two schemas that were assumed competing but turned out to be two layers of the same hypothesis. The decision layer (drawn primarily from the 25 March CSV/HTML) is outward-facing — what the client sees and acts on. The bucket layer (drawn primarily from the 28 April live register) is inward-facing — the noise-to-signal substrate aligned to R16. Both are required.

Section A is the spine that joins them. Section B is the decision layer. Section C is the bucket layer.

Design notes

R14 freeze

v5.0 is frozen. Schema changes require explicit version bump per ARCHITECTURE.md Section 19.8: doc version bump, version-comment note, dependent-tool updates, re-read on next session. Silent edits forbidden.

R16 alignment

Five canonical buckets: technology, cost, regulation, ecosystem, competitive. Section C names the four that are present and the one that is gapped. Geography sits as a documented exception, not a peer.

Source mix summary

  • 16 columns appear in both 25 March CSV and 28 April live with the same name
  • 5 columns in both, with rename — window / window_closes + window_date → window_status / window_closes_at; if_true / if_false → decision_if_true / decision_if_false; primary_sources_expected → primary_sources
  • 1 column from 25 March renamed in v5 — geography_tags → routing_geography
  • 4 columns from 28 April renamed in v5 — mkt_* → cost_*
  • 25 columns from 25 March only
  • 20 columns from 28 April only
  • 3 columns from 28 April kept as legacy (geo_*)
  • 6 columns new in v5 — register, schema_version, created_at, created_by, last_updated_by, plus type-promotion of last_updated

What v5 explicitly does not include

  • The per-metric hypothesis_metrics table — that ships with the trajectory layer (Build L) and carries the six R16 fields per metric
  • Junction tables for *_tags, company_tags, related_hyp_ids — parked to v6
  • Signal Tracker schema — Build E phase two
  • Any change to live data, the live Sheet, or n8n workflows
