EV charging infrastructure — structural ontology

Hierarchical taxonomy for signal matching. v0.1, 2026-05-03. Companion: ev_charging_decisions.md.

Structural decisions made (read first)

  1. Lifecycle phasing. Kept user's MAKE / BUILD / BUY / USE / MAINTAIN. BUY redefined as "commercial structures, capital flow, ownership models" rather than transactional procurement moments — procurement happens at every phase, so BUY-as-transactions fragments. As capital-and-commercial-structure it has a distinct shape.
  2. End-of-life folded into MAINTAIN (5.6), not a sixth phase. Decommissioning, recycling, second-life all live there — keeps user's 5-phase shape.
  3. Software is split across phases. Firmware + CSMS development under MAKE (1.4, 1.5); protocol operation under USE (4.5 OCPP, 4.3 ISO 15118); OTA under MAINTAIN (5.2). Same protocol appears in two places — leaves are different.
  4. AC/DC and public/private/depot/home are sub-axes, not top-level. They surface where they bite (connectors 1.2, grid connection 2.2), not as parallel branches.
  5. V2G / V2H / V2L sit under USE 4.3 alongside unidirectional energy delivery, with hardware enabler under MAKE 1.1.2. Deliberately scattered — V2G is multi-phase technology. Uncertain call (see decisions doc U2).
  6. Cross-cutting tags only. 10 axes flagged as xc:battery_storage etc.; parallel ontologies not built out per user instruction. Biggest cluster is power electronics 1.1 — almost every leaf shared with battery storage / grid mod / solar PV / motor drives.
  7. Depth target — bottoms out where the next level adds no signal-matching specificity. Power electronics → chip family with named vendors. Connectors → specific standard. Software → specific protocol version. Grid connection → kV class. Regulatory → specific instrument.
  8. Library — vanilla <details> + custom CSS, no CDN. Chosen over treant.js / d3 / jsTree because: zero dependencies (works offline), native semantics (keyboard + screen-reader), cannot break on JS error, mobile-friendly without responsive lib. JS only renders the tree, computes counts, populates leaf detail panel. ~120 lines.
